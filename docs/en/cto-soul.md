# CTO Soul Document

openCodex now splits the CTO soul into three layers:
- `prompts/cto-soul.md`: shared identity layer
- `prompts/cto-chat-soul.md`: chat / exploration / lightweight reply layer
- `prompts/cto-workflow-soul.md`: workflow orchestration layer

The three real child-agent lanes also have dedicated soul files:
- `prompts/cto-reply-agent-soul.md`
- `prompts/cto-planner-agent-soul.md`
- `prompts/cto-worker-agent-soul.md`

For a detached Telegram service install, openCodex prefers service-local copies at `<state-dir>/cto-soul.md`, `<state-dir>/cto-chat-soul.md`, and `<state-dir>/cto-workflow-soul.md` so the installed product does not depend on a repository checkout.

## Purpose

- define the long-lived CTO identity
- keep the default template grounded in the general-purpose Codex CLI personal assistant persona
- separate chat-mainline behavior from workflow-orchestration behavior
- keep Codex CLI as the local execution engine while openCodex stays thin
- state that the CTO identity belongs to the host-level supervisor, not to sandbox child sessions
- keep chat, discussion, and research as first-class CTO behaviors before orchestration when appropriate
- shape how the CTO infers intent, delegates work, and decides when to ask the CEO

## Current Behavior

- the CTO main thread loads the shared soul and then adds the chat/workflow overlay that matches the current mode
- detached Telegram services default those files to `<state-dir>/cto-soul.md`, `<state-dir>/cto-chat-soul.md`, and `<state-dir>/cto-workflow-soul.md`
- ad-hoc local runs still fall back to `prompts/cto-soul.md`, `prompts/cto-chat-soul.md`, and `prompts/cto-workflow-soul.md`
- the loaded documents are appended to the CTO main-thread system prompt in layers
- the host supervisor remains the CEO-facing CTO identity and workflow owner
- the CTO can stay in direct discussion/research mode instead of forcing every message into task orchestration
- sandbox child sessions are generated by the host supervisor as advisor, planner, reviewer, or helper sessions
- worker prompts are still generated by the CTO main thread, not by child agents
- if one layer is missing, openCodex falls back only for that layer
- the tray app can open the file directly through `Edit CTO Soul`
- the tray app can also open the mode overlays through `Edit CTO Chat Soul` and `Edit CTO Workflow Soul`
- the tray app can also restore the default template through `Restore Default CTO Soul`

## Editing Rules

- keep the content in English
- keep the instructions stable and role-oriented
- prefer high-level policy over task-specific commands
- use the shared layer for identity, language, safety, and product boundaries
- use the chat layer for natural replies, exploration, and "do not spawn a workflow yet" policy
- use the workflow layer for planning, delegation, waiting-question policy, and worker discipline
- use the reply-agent layer for the direct-reply helper's tone and escalation boundary
- use the planner-agent layer for the planning helper's decomposition style and confirmation threshold
- use the worker-agent layer for implementation helpers, rollback discipline, and reporting style
- use these files to shape the host-level CTO supervisor, not to store temporary task notes
