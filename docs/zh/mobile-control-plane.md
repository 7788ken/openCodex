# Mobile Control Plane

## 目的

本文定义 openCodex 下一阶段手机 / Web 入口的产品边界。

它不是一份实现方案，也不是命令规范补丁。
它只回答这些问题：

- 手机 / Web 入口能做什么
- 手机 / Web 入口不能做什么
- 默认应该通过什么方式暴露给手机
- 后续产品文档应如何面向普通用户表达部署与排障

## 核心定位

openCodex 的手机 / Web 入口是 control plane，不是第二套执行引擎，也不是远程 IDE。

它必须建立在现有这些边界之上：

- Codex CLI 仍然是本地执行引擎
- openCodex runtime 仍然持有工作流与 session 模型
- host supervisor 仍然持有主线程身份、状态归属与高风险控制权
- `remote`、`service`、`im` 仍然只是入口层，不替代 runtime 本体

这意味着手机入口只能成为宿主系统的一层窄控制面，不能反过来成为系统中心。

## 允许能力

第一阶段手机 / Web control plane 允许提供这些能力：

- 查看当前宿主状态
- 查看最近工作流、任务或消息摘要
- 发起聊天式控制输入
- 查看等待确认、排队中或已完成这类轻量状态
- 查看与当前入口直接相关的审计痕迹或最近活动

如果未来补 UI，这些能力都应保持“轻量、摘要、可读、可确认”的特点，而不是演化成完整工作台。

## 明确禁止

第一阶段手机 / Web control plane 不应提供这些能力：

- 远程桌面
- 完整 IDE 或文件树编辑器
- 在手机上直接拥有高风险执行能力
- 把浏览器登录态变成系统状态主轴

更直接地说：

- 手机入口可以发起请求
- 但不能篡改宿主权威状态
- 可以展示摘要
- 但不能代替本地工作界面

## 接入模型

默认接入模型应继续保持：

- local-first
- private-network-first
- 不默认公开暴露到公网

推荐原则：

- 本地服务优先只监听本机或受控地址
- 若要让手机访问，应优先放在用户自控的 VPN、私网或 tunnel 后面
- 不把托管公网 relay 当成第一阶段前提

这条原则与现有 `remote` 方向一致，但后续文档需要比当前更明确地把它放到默认推荐路径上。

## 面向普通用户的文档要求

如果后续把手机 / Web control plane 继续产品化，用户文档至少要补齐这些部分：

- 最短部署路径
- 成功判断标准
- 最常见失败点
- 首先应该查看哪些状态与日志

现有系统文档更偏架构、命令和运行时边界。
手机入口一旦增强，必须同步补齐面向普通用户的表达，而不是只留技术说明。

## 与现有能力的关系

### 与 `remote` 的关系

`remote` 仍然是第一阶段最小消息桥。
它当前解决的是“手机如何把消息发进来”，而不是“手机如何成为完整控制台”。

后续如果增强手机入口，仍应建立在这个边界之上：

- 先是消息桥
- 再是窄 control plane
- 不是直接变成远程 IDE

### 与 `im` 的关系

`im` 说明 openCodex 已经接受“手机入口不一定是网页”这个前提。
因此手机 / Web control plane 不是唯一远程入口，只是入口之一。

### 与 `service` / supervisor 的关系

手机入口不能拥有比宿主控制面更高的控制权。
状态归属和长期运行时状态，仍然应由宿主 `service` / supervisor 持有。

## 非目标

本文不定义：

- 具体前端技术栈
- 是否使用某个上游 Web UI
- 账户系统实现细节
- 完整的公网 relay 基础设施
- 原生 App 封装方案
- 具体命令参数变更

这些内容只有在产品边界稳定后才值得进入实现层。

## 收敛结论

openCodex 的手机 / Web 路线应继续收敛为：

- 一个建立在 runtime / supervisor 之上的窄 control plane
- 一个默认走私网 / 自控 tunnel 的本地优先方案

它不应收敛为：

- patched 上游 Web UI 产品
- 浏览器会话驱动的系统中心
- 远程 IDE
- 重依赖旁路运行环境的第二套安装形态
